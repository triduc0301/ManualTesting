# Giới thiệu
- **Kiểm thử hộp đen**: Tập trung vào input (đầu vào) và output (đầu ra), không quan tâm đến code bên trong. Phù hợp cho tester không cần kiến thức lập trình sâu, thường dùng ở mức System Testing và UAT.
- **Kiểm thử hộp trắng**: Tester xem xét cấu trúc code bên trong để kiểm tra logic và đường dẫn. Phù hợp cho developer, thường dùng ở mức Unit và Integration Testing.

|Tiêu chí|Hộp Đen|Hộp Trắng|
|-|-|-|
|Tập trung|Chức năng, yêu cầu|Code, logic nội bộ|
|Kiến thức cần|Yêu cầu người dùng|Mã nguồn, lập trình|
|Ưu điểm|Đơn giản, gần thực tế|Phát hiện lỗi sớm|
|Nhược điểm|Không phát hiện lỗi sâu|Tốn thời gian, cần code|

# Kiểm Thử Hộp Đen (Black-Box Testing)
Kiểm thử này giả định hệ thống như một "hộp đen" kín, chỉ kiểm tra hành vi bên ngoài. Ưu tiên thiết kế test case dựa trên yêu cầu.
### Các kỹ thuật chính
1. **Phân Vùng Tương Đương (Equivalence Partitioning)**:
    - Chia input thành các vùng tương đương (valid/invalid), nơi các giá trị trong vùng cho output giống nhau.
    - Test chỉ một đại diện mỗi vùng để tiết kiệm.
    - **Ví dụ**: Trường nhập tuổi (18-60). Vùng: <18 (invalid), 18-60 (valid), >60 (invalid). Test: 17, 30, 61.

2. **Phân Tích Giá Trị Biên (Boundary Value Analysis)**:
    - Test các giá trị biên (ranh giới) của vùng tương đương, vì lỗi thường xảy ra ở đây.
    - **Ví dụ**: Với tuổi 18-60, test: 17 (biên dưới invalid), 18 (biên dưới valid), 60 (biên trên valid), 61 (biên trên invalid).
3. **Bảng Quyết Định (Decision Table Testing)**:
    - Liệt kê tất cả tổ hợp input và output mong đợi để kiểm tra logic.
    - Ví dụ: Đăng nhập

|Username|Password|Output|
|-|-|-|
|Đúng|Đúng|Login thành công|
|Đúng|Sai|Lỗi password|
|Sai|Đúng|Lỗi username|
|Sai|Sai|Lỗi cả hai|

4. **Kỹ Thuật Đoán Lỗi (Error Guessing)**:
    - Dựa trên kinh nghiệm để dự đoán và test lỗi tiềm ẩn từ dự án trước.
    - Ví dụ: Test input đặc biệt như ký tự lạ, số âm, hoặc trường hợp mạng yếu.
5. **Chuyển Đổi Trạng Thái (State Transition Testing)**:
    - Kiểm tra sự thay đổi trạng thái theo sự kiện.
    - **Ví dụ**: Hệ thống đăng nhập:

|Trạng Thái Ban Đầu|Sự Kiện|Trạng Thái Mới|
|-|-|-|
|Logout|Nhập đúng user/pass|Login|
|Login|Nhấn logout|Logout|
|Login|Sai pass 3 lần|Khóa tài khoản|

6. **Kiểm Thử Thăm Dò (Exploratory Testing):**
- Không theo kịch bản cố định, tester tự khám phá dựa trên hiểu biết và kinh nghiệm.
- Khi nào dùng:
    - Tài liệu chưa đầy đủ.
    - Muốn phát hiện lỗi sáng tạo, tiềm ẩn.
- **Ví dụ**: Sử dụng app như người dùng thực, thử thao tác bất ngờ (back/forward nhanh).

# Kiểm Thử Hộp Trắng (White-Box Testing)
Kiểm thử này "mở hộp" để xem code bên trong, đảm bảo mọi đường dẫn và logic đều được kiểm tra.
### Các Kỹ Thuật Chính
1. **Đường Cơ Bản - Đồ Thị Dòng (Basis Path Testing - Control Flow Graph)**:
    - Vẽ đồ thị luồng điều khiển code để test tất cả đường dẫn độc lập.
    - **Ví dụ**: Với hàm if-else, test path true và false để bao quát 100% code.

2. **Dựa Trên Luồng Điều Khiển (Control Flow Testing)**:
    - Kiểm tra cấu trúc như if-else, loop, switch.
    - **Ví dụ**: Test vòng lặp for từ 0 đến n, đảm bảo không infinite loop.

3. **Dựa Trên Luồng Dữ Liệu (Data Flow Testing)**:
    - Theo dõi biến (define-use-kill): Định nghĩa, sử dụng, xóa.
    - **Ví dụ**: Biến x gán ở dòng 1, dùng ở dòng 5 → Kiểm tra không dùng biến chưa khởi tạo.

4. **Kiểm Thử Đột Biến (Mutation Testing)**:
    - Thay đổi code (mutants, ví dụ: + thành -) để xem test case có "giết" (phát hiện) mutant không.
    - **Ví dụ**: Nếu test case không phát hiện thay đổi, cần cải thiện test suite.
